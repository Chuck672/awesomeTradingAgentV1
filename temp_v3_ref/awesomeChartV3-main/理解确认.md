# 系统架构与数据集成理解确认

基于 `MT5数据集成方案设计_gemini.md` 和 `MT5数据集成方案设计_Claude.md` 的内容，以下是对系统核心设计的理解确认：

### 1. 冷热数据的边界是如何界定的？
**答：** 热数据存储在基于磁盘的 DuckDB 文件（如 `market.duckdb`）中，包含**当前未归档月份**的完整数据和实时追加的最新 K 线；冷数据则以高压缩比的 Parquet 格式按年/月分卷存储（如 `{year}_{month}.parquet`），包含**历史已归档的完整月份**数据。

### 2. MetaStore 在整个系统中的核心职责是什么？
**答：** MetaStore（如 `config.db` / `events.db`）作为系统的配置与状态中心，负责存储活跃的 `symbols` 列表、系统配置项，以及记录数据同步状态（如各品种最后同步时间戳、归档进度）。

### 3. 当用户打开一个新Symbol的图表时，系统的完整执行链路是什么？
**答：** 
1. **状态检查与归档**：检查该 Symbol 的 DuckDB 热数据，若存在跨月旧数据则触发异步 Parquet 归档并清理热库。
2. **断点续传**：查询热库最新时间戳，通过 MT5 API 增量拉取缺失的 K 线并写入热库。
3. **混合查询返回**：构建联合 SQL 查询 Parquet(冷) + DuckDB(热)，截断返回历史数据给前端渲染。
4. **建立实时通道**：通过 WebSocket 将后续从 ZMQ 接收到的实时 K 线/Tick 推送给图表。

### 4. GapDetector 的 analyze() 会产出哪几种 action 类型？
*(注：根据上下文推断及通用数据对齐逻辑)*
**答：** `GapDetector` 主要产出三种动作：**`FETCH_MISSING`**（检测到数据断层，需要向 MT5 API 拉取特定时间段的缺失数据）、**`ROLLOVER_ARCHIVE`**（检测到跨月/跨年，需要将热数据打包成 Parquet 冷存储）以及 **`NO_ACTION`**（数据连续，继续接收实时流）。

### 5. 实时数据的写入路径（从ZMQ到DuckDB）经过哪些环节？
**答：** 
MT5 EA 通过 **ZMQ PUB** 广播 JSON 消息 ➜ Python `MessageBus` (ZMQ SUB) 接收并解析为标准 `MarketMessage` ➜ 经过事件总线分发给数据摄入服务（可能附带事件溯源落盘 SQLite）➜ 最终由 `DuckDBManager` 执行 `INSERT OR REPLACE` 写入 `current_bar` 或 `ohlcv_hot` 表。
