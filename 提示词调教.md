1. 采用“多标签”分类而非“单选” (Multi-label Classification)
不要问 Supervisor “这是哪一类操作？”，而要问 “为了完成这个任务，需要启用哪些能力模块？”

逻辑变化：让 Supervisor 输出一个数组 required_capabilities: ["MARKET_DATA", "VISUAL_ANNOTATION"]。

实现方案：在 Prompt 中明确定义，如果用户提到“分析”、“寻找”、“对比”，必须强制包含 MARKET_DATA 模块。这样即使它识别出了“画图”，也会因为关联逻辑带上“数据”工具。

2. 引入“逻辑依赖链” (Logical Dependency Mapping)
在 Supervisor 的系统提示词中硬编码一套**“前置依赖规则”**。

Supervisor 系统指令示例：

你是 QuantLogicLab 的任务调度员。在分配任务前，请遵循以下依赖法则：

绘图依赖分析：任何涉及“画线”、“标记”、“高亮”的操作，若未指定确切坐标，必须同时开启 Data_Analyzer 权限以寻找价格点。

指标依赖行情：任何涉及“添加指标”的操作，必须开启 Chart_Control 权限。

交易依赖风控：任何涉及“执行下单”的操作，必须强制开启 Risk_Manager 权限。

3. 使用“思维链 (CoT)” 强制结构化输出
不要让模型直接给结果，让它先“复述”意图，再“拆解”步骤。这能显著提高模型对复杂句子的理解准确度。

推荐的输出 JSON 结构：

JSON
{
  "user_intent_analysis": "用户希望找到最近的高点并做标记",
  "logical_steps": [
    "1. 获取最近 100 根 K 线数据 (需要 DATA 模块)",
    "2. 识别最高价 (需要 ANALYZER 模块)",
    "3. 在该位置绘制水平线 (需要 DRAW 模块)"
  ],
  "active_tool_groups": ["data_fetcher", "technical_analyzer", "chart_painter"]
}
4. “少即是多”：Few-shot 的边际案例调教
在提示词中加入 3-5 个容易混淆的边缘案例 (Edge Cases)。

Case A (复合意图): “帮我看看现在的压力位并画出来。” -> 结果：[Data, Draw]

Case B (纯展示意图): “把图表缩放到刚才那一波上涨。” -> 结果：[View_Control]

Case C (条件交易): “如果突破刚才画的线就买入。” -> 结果：[Data, Trading, Trigger_Monitor]

5. 动态注入“当前上下文”作为决策辅助
Supervisor 的压力大，往往是因为它不知道“刚才发生了什么”。

优化策略：在发送给 Supervisor 的 Prompt 中，实时注入一个 Brief_State_Manifest（简要状态清单）：

当前屏幕是否有线？ (Is_Drawing_Active: true)

当前已选中的指标？ (Active_Indicators: ["SVP_v5"])

最近一次 AI 操作？ (Last_Action: "Calculated Resistance at 65000")

当用户说“在那儿画条线”时，Supervisor 看到 Last_Action，就能瞬间明白“那儿”指的是 65000，从而精准派发 draw_price_level(price=65000) 工具。

技术避坑指南：
避免使用模糊动词：在 Tool 说明中，把 analyze_data 改为 fetch_historical_ticks_for_calculation。名字越长、越具体，Supervisor 就越不容易调错。

容错设计：如果 Analyzer 发现自己需要 chart_tools 却没被授权，让它返回一个特殊的错误码 INSUFFICIENT_PERMISSION_FOR_TASK，触发 Supervisor 重新评估并补发权限。
